# Security

## Overview

Security in Asaas is implemented through multiple layers. While the API key serves as the foundational authentication mechanism, Asaas provides additional security features including IP whitelisting and transfer validation webhooks. These mechanisms work together to protect sensitive operations and prevent unauthorized access.

For mission-critical operations like transfers and withdrawals, combining multiple security layers is strongly recommended.

## API Key Management

### Generating API Keys

API keys can only be generated by admin users through the Asaas web interface. The platform does not provide programmatic key generation - all keys must be created manually via the dashboard.

**Key characteristics:**

- **Irrecoverable after generation** - Once you close the generation dialog, the key cannot be retrieved again. Store it securely immediately.
- **Maximum 10 keys per account** - Each account can maintain up to 10 active keys.
- **Named and dated** - Keys can be assigned descriptive names and optional expiration dates for better management.
- **Enable/disable without deletion** - Keys can be temporarily disabled and re-enabled without permanent deletion.
- **No restoration after deletion** - Once a key is deleted, it cannot be recovered or restored.

### Key Prefixes

API keys use environment-specific prefixes for operational diagnostics:

- Production keys: `$aact_prod_...`
- Sandbox keys: `$aact_hmlg_...`

While the SDK does not validate these prefixes for functionality, they are useful for debugging environment mismatches.

### Best Practices

**Use separate keys for different environments:**

```typescript
// Development
const sandboxClient = new AsaasClient({
  accessToken: process.env.ASAAS_SANDBOX_KEY,
  environment: 'SANDBOX',
})

// Production
const productionClient = new AsaasClient({
  accessToken: process.env.ASAAS_PRODUCTION_KEY,
  environment: 'PRODUCTION',
})
```

**Rotate keys periodically:**

Establish a key rotation schedule to minimize exposure risk. When rotating:

1. Generate a new key with an identifying name
2. Update your application configuration
3. Test the new key in a non-production environment
4. Deploy the configuration change
5. Disable (not delete) the old key
6. Monitor for errors before deleting the old key permanently

**Never hardcode keys:**

```typescript
// ❌ NEVER do this
const client = new AsaasClient({
  accessToken: '$aact_prod_abc123...',
  environment: 'PRODUCTION',
})

// ✅ Always use environment variables
const client = new AsaasClient({
  accessToken: process.env.ASAAS_ACCESS_TOKEN,
  environment: process.env.ASAAS_ENVIRONMENT === 'production' ? 'PRODUCTION' : 'SANDBOX',
})
```

**Store keys securely:**

- Use environment variables for runtime access
- Store production keys in secure vaults (AWS Secrets Manager, HashiCorp Vault, etc.)
- Never commit keys to version control
- Use `.env.local` or similar for local development (ensure it's in `.gitignore`)

## IP Whitelist

IP whitelisting restricts API access to requests originating from pre-authorized IP addresses. This adds a network-layer security control that protects against credential theft and unauthorized access.

### How It Works

When IP whitelisting is enabled:

1. All API requests must originate from whitelisted IPs
2. Requests from non-whitelisted IPs receive `HTTP 403 Forbidden`
3. The restriction applies to all API endpoints uniformly

### Configuration

IP whitelist configuration is managed exclusively through the Asaas dashboard:

1. Navigate to account security settings
2. Add individual IPs or small IP ranges
3. Save the whitelist configuration

Changes take effect immediately after saving.

### Important: Distinguishing 403 Errors

The SDK may encounter `HTTP 403` errors from two distinct sources:

1. **IP whitelist rejection** - Request originated from a non-whitelisted IP
2. **Permission denial** - Valid request but insufficient permissions for the operation

When debugging 403 errors, check:

- Whether IP whitelist is enabled in your Asaas account
- Whether your application's outbound IP matches the whitelist
- Whether the API key has the required permissions for the endpoint

### Auto-Approval for Whitelisted IPs

Transfers and withdrawals initiated from whitelisted IPs can be configured for automatic approval, bypassing manual authorization requirements. This is particularly useful for automated financial operations.

**Configuration:**

Enable auto-approval for whitelisted IPs in the Asaas dashboard security settings. Once enabled, transfers from whitelisted IPs will not require manual Token SMS or Token APP authorization.

**Use case:**

```typescript
// Transfer initiated from whitelisted IP with auto-approval enabled
const transfer = await asaas.transfers.createExternal({
  value: 1000.0,
  pixAddressKey: '12345678900',
  pixAddressKeyType: 'CPF',
})

// transfer.authorized will be true immediately
console.log(transfer.authorized) // true
```

## Transfer Validation Webhook

For additional security on transfers and withdrawals, Asaas provides a dedicated webhook validation mechanism. This allows your application to programmatically approve or refuse transfers based on custom business logic.

### How It Works

1. Your application creates a transfer via the API
2. Approximately 5 seconds later, Asaas sends a `POST` request to your configured webhook URL
3. Your webhook endpoint validates the transfer details
4. Your endpoint responds with `APPROVED` or `REFUSED`
5. Asaas processes or cancels the transfer based on the response

### Webhook Payload

The webhook payload mirrors the transfer creation response:

```typescript
{
  "id": "tra_000005048474",
  "type": "PIX",
  "dateCreated": "2026-03-13",
  "value": 1000.00,
  "netValue": 995.00,
  "status": "PENDING",
  "transferFee": 5.00,
  "operationType": "PIX",
  "pixAddressKey": "12345678900",
  "pixAddressKeyType": "CPF",
  "authorized": false,
  "externalReference": "order-12345"
}
```

### Authentication

Asaas authenticates webhook requests using a custom header:

```typescript
import { ASAAS_WEBHOOK_AUTH_HEADER } from 'asaas-sdk'

// ASAAS_WEBHOOK_AUTH_HEADER = 'asaas-access-token'

app.post('/webhooks/transfer-validation', (req, res) => {
  const token = req.headers[ASAAS_WEBHOOK_AUTH_HEADER]

  if (token !== process.env.ASAAS_WEBHOOK_TOKEN) {
    return res.status(401).send('Unauthorized')
  }

  const transfer = req.body

  // Your validation logic
  const isValid = validateTransfer(transfer)

  res.json({
    transferValidation: isValid ? 'APPROVED' : 'REFUSED',
  })
})
```

### Expected Responses

Your webhook must respond with a JSON object containing the `transferValidation` field:

**Approve the transfer:**

```json
{
  "transferValidation": "APPROVED"
}
```

**Refuse the transfer:**

```json
{
  "transferValidation": "REFUSED"
}
```

### Retry Logic

If your webhook endpoint fails to respond or returns an error:

- Asaas will attempt delivery up to **3 times**
- After 3 failed attempts, the transfer is **automatically cancelled**

Ensure your webhook endpoint is highly available and responds within a reasonable timeout.

### Implementation Example

```typescript
import { ASAAS_WEBHOOK_AUTH_HEADER } from 'asaas-sdk'
import express from 'express'

const app = express()
app.use(express.json())

app.post('/webhooks/transfer-validation', async (req, res) => {
  // 1. Authenticate the webhook
  const token = req.headers[ASAAS_WEBHOOK_AUTH_HEADER]
  if (token !== process.env.ASAAS_WEBHOOK_TOKEN) {
    return res.status(401).json({ error: 'Unauthorized' })
  }

  // 2. Parse and validate the transfer
  const transfer = req.body

  try {
    // 3. Apply your business rules
    const validationResult = await validateTransferBusinessRules(transfer)

    // 4. Check against fraud detection
    const isSuspicious = await checkFraudDetection(transfer)

    if (isSuspicious) {
      console.warn('Suspicious transfer detected:', transfer.id)
      return res.json({ transferValidation: 'REFUSED' })
    }

    // 5. Verify external reference matches expected state
    if (transfer.externalReference) {
      const orderValid = await verifyOrderState(transfer.externalReference)
      if (!orderValid) {
        return res.json({ transferValidation: 'REFUSED' })
      }
    }

    // 6. Respond with approval
    res.json({ transferValidation: 'APPROVED' })
  } catch (error) {
    console.error('Transfer validation error:', error)
    // Fail closed - refuse transfer on error
    res.json({ transferValidation: 'REFUSED' })
  }
})

  }
  return true
}

}
```

## Official Asaas IPs

Asaas maintains an official list of IP addresses used for webhook communication and platform services. This list is updated periodically and should be referenced from the official documentation rather than hardcoded.

**Official IP documentation:**
[https://docs.asaas.com/docs/official-asaas-ips](https://docs.asaas.com/docs/official-asaas-ips)

### Usage

If you need to whitelist Asaas IPs in your firewall or security groups:

1. Reference the official documentation page
2. Configure your infrastructure to allow traffic from listed IPs
3. Subscribe to Asaas changelog notifications to catch IP updates

**Do not hardcode IPs in your application code.** Treat the IP list as operational infrastructure configuration.

## HTTPS Requirement

When capturing credit card data through your own interface (rather than using Asaas-hosted payment pages), **HTTPS is mandatory**.

### When HTTPS is Required

- Custom checkout forms that collect card number, CVV, and expiration date
- Tokenization flows where card data passes through your servers
- Any direct handling of cardholder data

### PCI-DSS Compliance

Direct handling of credit card data requires PCI-DSS compliance. Using Asaas-hosted payment pages (transparent checkout) significantly reduces your PCI compliance scope by avoiding direct card data handling.

**Recommended approach:**

```typescript
// ✅ Use Asaas-hosted checkout (reduced PCI scope)
const payment = await asaas.payments.create({
  customer: 'cus_000005048474',
  billingType: 'CREDIT_CARD',
  value: 100.0,
  creditCard: {
    holderName: 'João Silva',
    // Card details collected via Asaas hosted page
  },
})

// ❌ Custom form requires HTTPS and PCI compliance
// Your custom form → Your server → Asaas
```

If you must implement custom card data capture, ensure:

- All pages use HTTPS with valid certificates
- No card data is logged or stored
- Tokenization happens immediately upon capture
- Your infrastructure meets PCI-DSS requirements for your compliance level

## Best Practices Summary

### Combined Security Layers

For production environments handling sensitive operations:

1. **Enable IP whitelist** for network-layer protection
2. **Configure transfer validation webhook** for programmatic approval
3. **Rotate API keys periodically** (quarterly or on security events)
4. **Monitor for 403 errors** that may indicate IP whitelist issues
5. **Use separate keys per environment** to isolate sandbox and production

### Monitoring

Implement monitoring for security-related events:

```typescript
import { AsaasClient, AsaasApiError } from 'asaas-sdk'

const client = new AsaasClient({
  accessToken: process.env.ASAAS_ACCESS_TOKEN!,
  environment: 'PRODUCTION',
})

try {
  await client.payments.create(paymentData)
} catch (error) {
  if (error instanceof AsaasApiError) {
    if (error.status === 401) {
      // Authentication failure - possible key compromise
      console.error('Authentication failed - check API key')
      alertSecurityTeam('API key authentication failure')
    } else if (error.status === 403) {
      // May indicate IP whitelist rejection
      console.error('Access forbidden - check IP whitelist')
      alertSecurityTeam('Possible IP whitelist rejection')
    } else if (error.status === 429) {
      // Rate limit exceeded - possible abuse
      console.warn('Rate limit exceeded')
      alertOperationsTeam('Rate limit threshold reached')
    }
  }
}
```

### Environment Variables

Use a consistent naming convention for environment variables:

```bash
# .env.production

# .env.sandbox

ASAAS_ENVIRONMENT=sandbox
ASAAS_WEBHOOK_TOKEN=<webhook-auth-token>
```

### Transfer Security Checklist

When implementing transfers and withdrawals:

- ✅ Enable IP whitelist for production environment
- ✅ Configure transfer validation webhook
- ✅ Implement robust webhook authentication
- ✅ Handle webhook delivery failures gracefully
- ✅ Monitor `authorized` field in transfer responses
- ✅ Document expected transfer approval flows
- ✅ Test both automatic and manual approval paths

### HTTPS Checklist

When handling payment card data:

- ✅ All pages use HTTPS with valid certificates
- ✅ Card data never logged or stored (except via Asaas tokenization)
- ✅ PCI-DSS compliance level determined and requirements met
- ✅ Security scanning and vulnerability assessment in place
- ✅ Consider using Asaas-hosted pages to reduce PCI scope

## Security Incident Response

If you suspect a security incident:

1. **Immediately disable the affected API key** via Asaas dashboard
2. **Generate a new key** and update your application configuration
3. **Review access logs** for unauthorized activity
4. **Update IP whitelist** if compromise involved network access
5. **Rotate webhook authentication tokens**
6. **Contact Asaas support** if the incident involves platform security
