# Subscriptions

The Subscriptions service (`client.subscriptions`) manages recurring payment subscriptions, allowing you to charge customers periodically. This guide covers subscription lifecycle management, credit card handling, payment tracking, and invoice settings configuration.

## Key Concepts

### Cycle Types

Subscriptions support multiple billing cycles:

- `WEEKLY` - Charge every 7 days
- `BIWEEKLY` - Charge every 14 days
- `MONTHLY` - Charge every 30 days
- `BIMONTHLY` - Charge every 60 days
- `QUARTERLY` - Charge every 90 days
- `SEMIANNUALLY` - Charge every 180 days
- `YEARLY` - Charge every 365 days

### Status Lifecycle

Subscriptions progress through the following statuses:

- `ACTIVE` - Subscription is generating charges
- `INACTIVE` - Subscription is paused and not generating charges
- `EXPIRED` - Subscription has ended (reached `endDate` or `maxPayments`)

You can update the status between `ACTIVE` and `INACTIVE` to pause or resume a subscription.

### Billing Types

When creating a subscription, you can specify:

- `UNDEFINED` - Customer chooses payment method
- `BOLETO` - Bank slip (boleto bancario)
- `CREDIT_CARD` - Credit card
- `PIX` - Instant payment

When reading subscription data, additional types may be returned:

- `DEBIT_CARD` - Debit card
- `TRANSFER` - Bank transfer
- `DEPOSIT` - Deposit

### Credit Card Subscription Flow

For subscriptions paid by credit card, use the dedicated `createWithCreditCard()` method. This method:

1. Creates the subscription
2. Tokenizes the credit card
3. Processes the first charge immediately
4. Stores the card for future recurring charges

To update the card without charging, use `updateCreditCard()`. This updates both the subscription and any pending charges linked to it.

### Split Blocking Rule

When using split in subscriptions, Asaas validates that the total split amount does not exceed the net receivable value. If validation fails:

1. The subscription is blocked
2. Split is disabled automatically
3. No new recurring charges are generated

This prevents over-allocation of funds across split recipients.

### Soft Delete

Subscriptions use soft deletion. Use `remove()` to mark as deleted. Unlike customers and payments, **subscriptions do not have a restore endpoint** in the current API version.

## Subscriptions Service

### Methods Reference

| Method | Description |
|--------|-------------|
| `create(params)` | Create a new subscription |
| `createWithCreditCard(params)` | Create subscription with credit card data |
| `get(id)` | Retrieve a subscription by ID |
| `list(params?)` | List subscriptions with filters (paginated) |
| `update(id, params)` | Update an existing subscription |
| `updateCreditCard(id, params)` | Update credit card without charging |
| `remove(id)` | Soft-delete a subscription |
| `listPayments(id, params?)` | List payments generated by subscription (paginated) |
| `downloadPaymentBook(id)` | Download subscription payment book PDF |
| `listInvoices(id, params?)` | List invoices for subscription charges (paginated) |

### Examples

#### Create a Monthly Subscription

```typescript
import { AsaasClient } from '@rodrigogs/asaas-sdk';

const client = new AsaasClient({
  accessToken: process.env.ASAAS_ACCESS_TOKEN!,
  environment: 'SANDBOX',
});

const subscription = await client.subscriptions.create({
  customer: 'cus_000005836142',
  billingType: 'BOLETO',
  value: 99.90,
  nextDueDate: '2026-04-01',
  cycle: 'MONTHLY',
  description: 'Premium Plan - Monthly Subscription',
  externalReference: 'SUB-2026-001',
});

console.log('Subscription ID:', subscription.id);
console.log('Status:', subscription.status);
console.log('Next due date:', subscription.nextDueDate);
```

const creditCardSubscription = await client.subscriptions.createWithCreditCard({

#### Update Subscription Details

```typescript
const updated = await client.subscriptions.update(subscription.id, {
  value: 119.90,
  description: 'Premium Plan - Updated pricing',
  cycle: 'QUARTERLY',
  updatePendingPayments: true, // Apply changes to pending charges
});

console.log('Updated value:', updated.value);
console.log('Updated cycle:', updated.cycle);
```

#### Pause and Resume Subscription

```typescript
// Pause subscription (stop generating charges)
await client.subscriptions.update(subscription.id, {
  status: 'INACTIVE',
});

console.log('Subscription paused');

// Resume subscription
await client.subscriptions.update(subscription.id, {
  status: 'ACTIVE',
});

console.log('Subscription resumed');
```

#### List Subscription Payments

```typescript
// List all payments generated by this subscription
const result = await client.subscriptions.listPayments(subscription.id, {
  status: 'PENDING',
});

for await (const payment of result) {
  console.log('Payment ID:', payment.id);
  console.log('  Due date:', payment.dueDate);
  console.log('  Value:', payment.value);
  console.log('  Status:', payment.status);
  console.log('  Billing type:', payment.billingType);
}
```

#### Filter Subscription List

```typescript
// List active subscriptions for a specific customer
const activeSubscriptions = await client.subscriptions.list({
  customer: 'cus_000005836142',
  status: 'ACTIVE',
  billingType: 'CREDIT_CARD',
  limit: 20,
});

for await (const sub of activeSubscriptions) {
  console.log(`Subscription ${sub.id}:`);
  console.log(`  Value: R$ ${sub.value}`);
  console.log(`  Cycle: ${sub.cycle}`);
  console.log(`  Next due: ${sub.nextDueDate}`);
}
```

#### Download Payment Book

The payment book is a PDF containing all payment slips for the subscription.

```typescript
import { writeFile } from 'fs/promises';

const paymentBook = await client.subscriptions.downloadPaymentBook(subscription.id);

// BinaryResponse contains: data (Buffer), filename, contentType
await writeFile(
  paymentBook.filename || 'subscription-payment-book.pdf',
  paymentBook.data
);

console.log('Payment book saved:', paymentBook.filename);
console.log('Content type:', paymentBook.contentType);
console.log('File size:', paymentBook.data.length, 'bytes');
```

## Invoice Settings

The invoice settings sub-service (`client.subscriptions.invoiceSettings`) configures automatic fiscal invoice (nota fiscal) generation for subscription charges.

### Methods Reference

| Method | Description |
|--------|-------------|
| `create(subscriptionId, params)` | Create invoice settings for subscription |
| `get(subscriptionId)` | Retrieve invoice settings |
| `update(subscriptionId, params)` | Update invoice settings |
| `remove(subscriptionId)` | Remove invoice settings |

### Examples

#### Configure Automatic Invoice Generation

```typescript
const invoiceSettings = await client.subscriptions.invoiceSettings.create(
  subscription.id,
  {
    municipalServiceId: '1234',
    municipalServiceCode: '01.07',
    municipalServiceName: 'Desenvolvimento de software sob encomenda',
    effectiveDatePeriod: 'ON_PAYMENT_CONFIRMATION',
    receivedOnly: true,
    observations: 'Serviço de desenvolvimento de software personalizado',
    taxes: {
      retainIss: false,
      iss: 2.00,
      cofins: 3.00,
      csll: 1.00,
      inss: 0.00,
      ir: 0.00,
      pis: 0.65,
    },
  }
);

console.log('Invoice settings created');
console.log('Municipal service:', invoiceSettings.municipalServiceName);
```

#### Get Invoice Settings

```typescript
const settings = await client.subscriptions.invoiceSettings.get(subscription.id);

console.log('Municipal service ID:', settings.municipalServiceId);
console.log('Invoice creation period:', settings.invoiceCreationPeriod);
console.log('Taxes:', settings.taxes);

```

#### Update Invoice Settings

```typescript
await client.subscriptions.invoiceSettings.remove(subscription.id);

```

### Invoice Effective Date Periods

The `effectiveDatePeriod` parameter controls when invoices are issued:

- `ON_PAYMENT_CONFIRMATION` - Generate invoice when payment is confirmed
- `ON_PAYMENT_DUE_DATE` - Generate invoice on the due date
- `BEFORE_PAYMENT_DUE_DATE` - Generate invoice before due date (use with `daysBeforeDueDate`)
- `ON_DUE_DATE_MONTH` - Generate invoice on the due date month
- `ON_NEXT_MONTH` - Generate invoice in the next month

**Important**: The request parameter is `effectiveDatePeriod`, but the response field is `invoiceCreationPeriod`. This is an API quirk that the SDK preserves as-is.

### Tax Configuration

The `taxes` object supports the following Brazilian tax types:

- `retainIss` - Whether to retain ISS tax (boolean)
- `iss` - ISS percentage (0-100)
- `cofins` - COFINS percentage
- `csll` - CSLL percentage
- `inss` - INSS percentage
- `ir` - IR percentage
- `pis` - PIS percentage

All tax percentages are decimal values (e.g., `2.00` for 2%).

## Sandbox Limitations

The sandbox environment fully supports subscription testing with the following notes:

### Generating New Charges

According to Asaas documentation, **to test the generation of new subscription charges in sandbox**, you must download the payment book using `downloadPaymentBook()`. This triggers the generation of the next charge in the sequence.

```typescript
// Trigger next charge generation in sandbox
const paymentBook = await client.subscriptions.downloadPaymentBook(subscription.id);
console.log('Payment book generated - next charge should be created');

// Wait a moment, then check for new payments
const payments = await client.subscriptions.listPayments(subscription.id);
console.log('Total payments generated:', payments.totalCount);
```

### Fully Testable Features

- Subscription creation (all billing types)
- Credit card subscriptions and card updates
- Subscription updates and status changes
- Soft deletion
- Split configuration
- Invoice settings configuration
- Payment listing

### Not Testable in Sandbox

The following features cannot be fully tested in sandbox and require production validation:

- Actual recurring charge generation on schedule (use payment book download to simulate)
- Real invoice issuance with tax authorities
- Split blocking validation in production scenarios

## Best Practices

1. **Always use `createWithCreditCard()` for card subscriptions** rather than mixing create and payment methods
2. **Set `updatePendingPayments: true`** when updating subscription values to apply changes to pending charges
3. **Monitor split allocations** to avoid subscription blocking due to over-allocation
4. **Use `externalReference`** to link subscriptions to your internal billing system
5. **Configure invoice settings early** if you need automatic nota fiscal generation
6. **Test charge generation** in sandbox using the payment book download technique
7. **Use `endDate` or `maxPayments`** to create time-limited subscriptions
8. **Pause subscriptions** with `status: INACTIVE` rather than deleting when temporary suspension is needed
9. **List payments regularly** to track subscription charge history and status
10. **Handle split blocking errors** gracefully by adjusting split configuration or subscription value
